# Model Configuration Guide

This guide describes how model entries work in the current `niffler-nats` branch.

## Where Models Are Configured

Models are configured in `config.yaml`:

- macOS/Linux: `~/.niffler/config.yaml`
- Windows: `%APPDATA%/niffler/config.yaml`

The default template generated by `niffler init` contains working examples for:

- OpenRouter
- Ollama
- LM Studio
- additional remote OpenAI-compatible providers

## Basic Model Example

```yaml
models:
  - nickname: "openrouter-free"
    base_url: "https://openrouter.ai/api/v1"
    model: "qwen/qwen3-coder:free"
    context: 262144
    enabled: true
    type: "standard"
    api_env_var: "OPENROUTER_API_KEY"
    temperature: 0.7
    top_p: 0.95
    max_tokens: 32768
```

## Common Fields

### Required In Practice

- `nickname`: short local model name used with `--model`
- `base_url`: OpenAI-compatible API endpoint
- `model`: provider model identifier
- `context`: context window size
- `enabled`: whether the model is selectable

### Common Optional Fields

- `type`: protocol type, usually `standard`
- `api_key`: explicit API key in config
- `api_env_var`: environment variable name for the API key
- `temperature`
- `top_p`
- `max_tokens`
- `reasoning`
- `input_cost_per_mtoken`
- `output_cost_per_mtoken`

## Credential Resolution

For a model, Niffler looks for credentials in this order:

1. `api_key` in `config.yaml`
2. key storage file used by Niffler
3. the environment variable named by `api_env_var`

Placeholder values such as `ENTER YOUR ...` are treated as placeholders, not real credentials.

If both a real `api_key` and `api_env_var` are present, the explicit `api_key` wins.

## Local Models

Local OpenAI-compatible servers do not require real remote credentials.

### Ollama

Typical endpoint:

```yaml
base_url: "http://localhost:11434/v1"
```

### LM Studio

Typical endpoint:

```yaml
base_url: "http://localhost:1234/v1"
```

For local servers, a placeholder non-empty key is acceptable if the server expects an `Authorization` header.

## Remote Models

For remote APIs such as OpenRouter or Synthetic, you need usable credentials if the model is enabled.

At startup, Niffler warns about enabled remote models that are missing usable credentials.

## OpenRouter Example

```bash
export OPENROUTER_API_KEY="your-key-here"
```

```yaml
models:
  - nickname: "openrouter-nemotron-free"
    base_url: "https://openrouter.ai/api/v1"
    model: "nvidia/nemotron-3-super-120b-a12b:free"
    context: 131072
    enabled: true
    type: "standard"
    api_env_var: "OPENROUTER_API_KEY"
```

## Selecting A Model

Use a model by nickname:

```bash
niffler --model=openrouter-free
```

List configured models:

```bash
niffler model list
```

## Cost Tracking

If `input_cost_per_mtoken` and `output_cost_per_mtoken` are set, Niffler can estimate and record usage cost.
